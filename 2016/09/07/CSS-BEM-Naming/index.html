CSS BEM 命名规范

09/07/2016 CSS

BEM简介

什么的BEM？

BEM是Block-Element-Modifier的缩写，是由Yandex公司推出的CSS命名规范，官方是这么描述的：

  • BEM是一种命名方法，能够帮助你在前端开发中实现可复用的组件和代码共享。

Block

BEM中的Block并不是 inline-block 中的block，而是逻辑上和功能上相对独立的组件，也就是我们常说的模块。

Block的特点：

  • 实现嵌套；
  • 相对独立；
  • 可重用。

下面一一说明：
一个Block中可以嵌套其他的Block，如下图，一个复杂的 head block 由 menu block 、 auth block 、 logo block 、 search block四个子模块共同组成。一个块是一个独立的实体，就像应用的一块“积木”，一个块既可以是简单的也可以是复合的。

Block是相对独立的，可以出现在页面中的任意位置，可以在不同页面甚至不同项目中运行良好。如下图，验证模块可以在左边也可以在右边。

Block具有可重用性。如下图，一个页面中可以出现多个相同的Block。

Element

元素是块的一部分，如下图，menu block 由不同的 menu Element 组成。

这里有一个误区：如果一个Element-son是另一个Element-father的子元素，那么写法应该是 Block__Element-father__Element-son，这样如果层级比较深的话，名字会很长。

其实不是的，正确的思路是：一个Block下的所有的Element无论相互层级如何，都要摊开属于Block，所以 BEM 最多只有 B+E+M 三级,不可能出现 B+E+E+..+E+M 超长class名,也要求E不能同名。

Modifier

修饰名即我们经常写的.active 、.current等表达状态。

BEM能解决什么问题？

css的样式应用是全局性的，没有作用域可言。考虑以下场景

场景一：开发一个弹窗组件，在现有页面中测试都没问题，一段时间后，新需求新页面，该页面一打开这个弹窗组件，页面中样式都变样了，一查问题，原来是弹窗组件和该页面的样式相互覆盖了，接下来就是修改覆盖样式的选择器…又一段时间，又开发新页面，每次为元素命名都心惊胆战，求神拜佛，没写一条样式，F5都按多几次，每个组件都测试一遍…

场景二：承接上文，由于页面和弹窗样式冲突了，所以把页面的冲突样式的选择器加上一些结构逻辑，比如子选择器、标签选择器，借此让选择器独一无二。一段时间后，新同事接手跟进需求，对样式进行修改，由于选择器是一连串的结构逻辑，看不过来，嫌麻烦，就干脆在样式文件最后用另一套选择器，加上了覆盖样式…接下来又有新的需求…最后的结果，一个元素对应多套样式，遍布整个样式文件…

以往开发组件，我们都用“重名概率小”或者干脆起个“当时认为是独一无二的名字”来保证样式不冲突，这是不可靠的。

理想的状态下，我们开发一套组件的过程中，我们应该可以随意的为其中元素进行命名，而不必担心它是否与组件以外的样式发生冲突。

BEM解决这一问题的思路在于，由于项目开发中，每个组件都是唯一无二的，其名字也是独一无二的，组件内部元素的名字都加上组件名，并用元素的名字作为选择器，自然组件内的样式就不会与组件外的样式冲突了。

这是通过组件名的唯一性来保证选择器的唯一性，从而保证样式不会污染到组件外。
